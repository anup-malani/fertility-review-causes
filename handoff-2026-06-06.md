## What this project is

A Cochrane-style systematic review evaluating every major proposed explanation for fertility decline. Each hypothesis is sorted into one of four categories (demographic, economic, biological, cultural) and evaluated against three target phenomena (pre-modern fertility variation, First Demographic Transition, Second Demographic Transition). Per hypothesis we produce a per-phenomenon GRADE rating of **causal credibility** and a **demographic-significance** verdict, synthesized into one chapter per hypothesis. Sequential deliverables: chapters → book → JEL summary article.

The full methodology is in [`PROTOCOL.md`](./PROTOCOL.md). RA operating manual in [`RA-PLAYBOOK.md`](./RA-PLAYBOOK.md). Master hypothesis list (currently an empty stub) in [`HYPOTHESES.md`](./HYPOTHESES.md). Plan that generated all of this: `/Users/amalani/.claude/plans/cozy-growing-crab.md`.

## Operating principle

**Every artifact comes from a `Workflow` invocation.** RAs invoke workflows; they do not run ad-hoc agent prompts. The workflow scripts live under `.claude/workflows/`. The 13 pipeline workflows (`enumerate-hypotheses.mjs` through `cross-chapter-check.mjs` + `pilot.mjs`) are currently **stubs that throw on invocation** — so an RA can't accidentally run one before it's implemented.

## Next session — implement `enumerate-hypotheses.mjs`

Goal: produce a draft `HYPOTHESES.md` for PI review by fanning out 4 parallel agents (one per category) to propose candidate hypotheses, then merging with cross-category resolution.

### Design sketch

```
parallel agents (one per category) → per-category hypothesis lists → merge step → draft HYPOTHESES.md
```

Per-category agent prompt should:
- Take a one-paragraph category brief from PROTOCOL.md §3.
- Read PROTOCOL.md §2 (phenomena) and §4 (operational definitions) for context — so the agent knows the hypotheses will be evaluated against three phenomena and needs to flag which it plausibly speaks to.
- Return a structured list of candidate hypotheses, each with:
  - `slug` (kebab-case, used as filename later, e.g., `quantity-quality-tradeoff`)
  - `name` (display name)
  - `category` (Demographic / Economic / Biological / Cultural)
  - `one_sentence_claim` (the precise causal claim)
  - `phenomena_relevance` (subset of {PM, FDT, SDT})
  - `seminal_citations` (3–5 anchor papers — author-year)
  - `cross_category_flag` (if it spans categories, name the secondary)
  - `notes` (anything the PI should know when approving)

### Lessons learned from `scaffold.mjs`

The first scaffold workflow failed because I put a `schema:` constraint on a file-scaffolding agent that did 12 tool calls but never issued the final `StructuredOutput` call. Two takeaways for future workflow scripts:

1. **Use `schema:` only when you actually need parsed output downstream.** If verification can be done by reading the filesystem (or by looking at the output directly), skip the schema and let the agent return free text.
2. **When you do use a schema, make the agent prompt's last instruction `"return the StructuredOutput tool call now"` explicit.** Models sometimes get distracted finishing the work and forget to issue the structured-output call.

For `enumerate-hypotheses.mjs`, a schema **is** useful because the output feeds directly into a merge step — but keep each per-category agent's schema small (a list of hypothesis records) and end the prompt with an explicit final-call instruction.

### Recovery pattern if a workflow fails mid-run

Workflow's resume cache only stores **successful** `agent()` calls. A failed call is not cached, so `resumeFromRunId` re-runs it from scratch — useful when you edited downstream stages but not when you need to rescue partial work from the failed stage.

If a workflow does partial work before failing:
- Check the project dir for what got written.
- If the missing pieces are well-defined static content, finish them directly with Write/Edit — don't relaunch the workflow just to keep purity.
- For the remaining agent-judgment parts, write a smaller focused workflow that does only the missing work. This is how `scaffold.mjs` was recovered (split into static-files-direct + `wire-git-fertility-explanations-review.mjs` for git wiring).

### Open design questions to resolve when implementing

- **Concurrency model** — 4 parallel category agents is the obvious shape, but should each category have a single proposer or N parallel proposers with a dedup step? For enumeration where coverage matters more than speed, two parallel proposers per category with a merge would catch hypotheses one proposer might miss. Tradeoff: 2× cost. Default to one per category for the first run; expand if HYPOTHESES.md feels thin after PI review.
- **PI approval gate** — does the workflow write a draft `HYPOTHESES.md` and stop, or does it prompt the PI inline? Recommend write-and-stop. The PI reviews the file in the repo, edits down/up as needed, commits, then invokes the next workflow. This keeps the workflow stateless and the PI's role clear.
- **Source pool for proposed hypotheses** — should the agent draw from its training-time knowledge of the demography literature, or do a search pass first to ground proposals in the actual literature? Recommend training-time knowledge for v1 (faster, sufficient for an initial enumeration), with a "completeness critic" agent as a follow-up phase that does a literature search and asks "what proposed explanations did the four agents miss?" That's a second invocation, easy to add after the v1 list is in PI hands.
